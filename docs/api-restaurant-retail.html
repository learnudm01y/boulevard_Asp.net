Boulevard API — Restaurant, Retail, Laundry & Photography

Mobile Developer Reference  ·  All endpoints use GET  ·  All responses are JSON

Base URL: https://boulevard.r-y-x.net   |   All responses return: { "result": [...], "code": 200, "message": "success", "isSuccess": true }
ℹ️ Every response has the same wrapper: result = the data, isSuccess = true/false, code = 200 (success) or 406 (error). Always check isSuccess before using result.

⚠️ Note: category-products and brand-products return result as a single object (category/brand info + nested products: [...] array), not as a plain array.

ICV Boulevard Score Field

FieldTypeSourceDescription
icvBoulevardScore string | null Excel column AE (31st col) — "ICV Boulevard Score" In-Country Value score for the Boulevard platform (e.g. "100%"). Populated at bulk import time via the Admin → Add Bulk Products Excel upload. Returns null or empty string when not set.

Origin Field (Social Impact)

FieldTypeSourceDescription
origin string | null Excel column "Origin" — optional Product origin for Social Impact tracking (e.g. "Local", "Imported"). Populated at bulk import time via the Admin → Add Bulk Products Excel upload for all four portals: Grocery, Dessert & Flowers, Restaurant, Retail. Returns null or empty string when not set.

Pipeline — how the value reaches the response

  1. Admin uploads an Excel file through any of the four portals (Grocery / Dessert & Flower / Retail / Restaurant) via Admin/Product/AddBulk.
  2. ProductController.AddBulk reads the optional columns "ICV Boulevard Score" and "Origin" (header-name matched — case-insensitive, whitespace-tolerant) using OleDB and writes them as XML attributes icvBoulevardScore and origin in the intermediate XML string.
  3. TempProductDataAccess.AddTempProduct() parses the XML attributes and stores them in dbo.TempProducts.IcvBoulevardScore (NVARCHAR 50) and dbo.TempProducts.Origin (NVARCHAR 100).
  4. TempProductDataAccess.AddProduct() copies the values to dbo.Products.IcvBoulevardScore and dbo.Products.Origin.
  5. The Web API method ProductServiceAccess.getSmallDetailsProducts() maps both fields to ProductSmallDetailsResponse, which is serialised as icvBoulevardScore and origin in the JSON response.
  6. The mobile UI (grocery-mobile.html) reads p.icvBoulevardScore and p.origin from the response and renders corresponding badges on the product card.
Note: Both columns are optional in the Excel template. Existing files that do not include them will continue to import normally — the fields will simply be empty. Columns are detected by header name (case-insensitive, leading/trailing spaces ignored), not by position.

⚙️ SFC

Service Form Configuration — Dynamic Field Mapping API

Purpose: These endpoints expose the dynamic form schema configured by the admin for any service type (e.g. a specific Typing center). The Flutter developer can consume the JSON response to render the entire data-entry screen — sections, fields, options, attachment rules and validation — without any hardcoded form logic.

Flow: Call /api/v1/service-form/list to discover configured service types → use the serviceTypeKey from that response to call /api/v1/service-form/schema/{serviceTypeKey}.

Admin Panel: GET /admin/service-form-config — accessible via Typing → Typing Form Config in the sidebar. Forms are configured per service type (individual providers).
GET /api/v1/service-form/schema/{serviceTypeKey}

Returns the full form schema for the given service type — sections, fields, options, attachment rules, validation metadata — everything Flutter needs to render the screen dynamically. Use the serviceTypeKey from the /api/v1/service-form/list endpoint or from the SearchingServiceType / GetServices endpoints.

GET /api/v1/service-form/list

Returns a lightweight list of all service types that have form configurations. Use this to discover which service types support dynamic form rendering. Pass the serviceTypeKey to /api/v1/service-form/schema/{serviceTypeKey}.

Database Tables:
  • ServiceFormSections — section groups per ServiceType (title, order, active flag) — FK: ServiceTypeId
  • ServiceFormFields — fields inside a section (key, label, type, data type, validation, etc.)
  • ServiceFormFieldOptions — choices for dropdown / radio / checkbox fields
  • ServiceFormAttachmentRules — file upload constraints per file-type field
